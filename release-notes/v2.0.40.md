# TankAlarm v2.0.40

**Theme: OTA version ordering — "Not newer than running firmware" fix.**

This release fixes the firmware-version comparison used by the over-the-air (OTA) update guard so a
new release is correctly recognized as *newer* than the running firmware. It builds directly on the
OTA reliability work delivered across v1.9.30–v1.9.42 and the dashboard-integrity fixes in v2.0.0.

---

## The bug

`tankalarm_versionToSeq()` packed a semantic version into a single integer with the formula
`major*100 + minor*10 + patch`. That is only monotonic while `minor <= 9` **and** `patch <= 9`. Once
the patch number grew past 9 it overflowed into the minor/major place values:

| Version  | Old sequence |
|----------|--------------|
| 1.9.42   | `1*100 + 9*10 + 42` = **232** |
| 2.0.0    | `2*100 + 0*10 + 0`  = **200** |

So a client running `1.9.42` computed `200 <= 232` and **refused** the `2.0.0` OTA with
*"not newer than running firmware."* The download/staging path itself was already working (fixed by
the `/fs` → `/cfg` mount-name change in v1.9.41); only the version *comparison* was blocking.

A second, related defect: the downgrade guard compared `versionToSeq(offered)` against
`FIRMWARE_BUILD_SEQ` — a separate monotonic *build counter* that only happened to coincide with the
version sequence through the 1.9.x line.

---

## The fix

- **Monotonic version sequence.** `tankalarm_versionToSeq()` now uses
  `major*1000000 + minor*1000 + patch`, giving each component up to 999 of head-room so a large patch
  number can never overflow into the next field. `2.0.40` → `2000040` is correctly greater than
  `1.9.42` → `1009042`.
- **Guard compares like with like.** The client downgrade guard now compares the offered version
  against `versionToSeq(FIRMWARE_VERSION)` (the running version) instead of `FIRMWARE_BUILD_SEQ`.
- **Migration-safe trial/rollback check.** `tankalarm_resolvePendingOta()` now decides trial-success
  vs. rollback by comparing the running version against the **staged target version string**
  (`target_v`), not the stored `target_seq`. Because `pending_ota.json` is written by the *old*
  firmware (in the old sequence scale) and read by the *new* firmware, comparing stored sequence
  numbers across this formula change would have false-triggered a rollback; comparing versions is
  scale-independent.

---

## Why 2.0.40 (and not 2.0.1)

Under the **old** (still-running) `1.9.42` guard, `versionToSeq("2.0.40")` = `240`, which *is*
greater than `232`, so a `1.9.42` device will accept the `2.0.40` OTA without a USB re-flash. Once it
is running `2.0.40`, the corrected formula is in place and all future comparisons are monotonic.

## Files changed
- `TankAlarm-112025-Common/src/TankAlarm_DFU.h` — monotonic `versionToSeq`; version-based
  trial/rollback resolution.
- `TankAlarm-112025-Client-BluesOpta/TankAlarm-112025-Client-BluesOpta.ino` — downgrade guard
  compares against `versionToSeq(FIRMWARE_VERSION)`.
- `TankAlarm-112025-Common/src/TankAlarm_Common.h` — `FIRMWARE_VERSION 2.0.40`, `FIRMWARE_BUILD_SEQ 234`.
- `TankAlarm-112025-Common/library.properties` — `version=2.0.40`.

## Deployment
After CI builds the release, assign the signed `TankAlarm-Client-secure-v2.0.40.slot.bin` image on
Notehub. The running `1.9.42` client will accept it, update to `2.0.40`, and from then on evaluate
every OTA with the corrected, monotonic version ordering.
