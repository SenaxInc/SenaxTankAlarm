# TankAlarm v2.0.41

**Theme: fix the client OTA image build so MCUboot can actually boot it.**

This is a build/release fix on top of v2.0.40. The firmware logic is unchanged from v2.0.40 (the
monotonic `versionToSeq` OTA-ordering fix); what changed is **how the client OTA slot image is
compiled and signed** in CI.

---

## The bug

When v2.0.40 was offered over the air, the device **accepted, downloaded, and staged** it
correctly (the v2.0.40 version-ordering fix worked), and MCUboot scheduled the swap. But the trial
boot was **reverted by MCUboot**, and the device fell back to the previous firmware:

> `OTA reverted (1.9.42->2.0.40): trial boot reverted by MCUboot`

Root cause was a regression in the release workflow: the *"build the client OTA slot image with the
Arduino core's `security=sien` setting"* step had been silently replaced with an older
"build-plain + external imgtool + custom keys" approach. That produced a slot image that was:

- **Linked for the wrong address** (`0x08040000`, a direct-boot layout) instead of the MCUboot slot
  address (`0xA0000000`), and
- **Signed with keys the Opta bootloader does not trust** (a project-local keychain instead of the
  core's bundled MCUboot keychain).

So when MCUboot swapped the image in and jumped to it, it faulted immediately — before the firmware
could confirm itself healthy — and the bootloader rolled back.

## The fix

- The release workflow again builds the client OTA image with
  `arduino-cli compile --fqbn arduino:mbed_opta:opta:security=sien` (plus `build.version=X.Y.Z+0`).
  This links the firmware for the MCUboot slot address and signs/encrypts it with the core's bundled
  imgtool and the default MCUboot keychain the Opta bootloader actually trusts — the format proven to
  validate and swap on hardware.

## Why 2.0.41 (not a rebuild of 2.0.40)

After the reverted trial boot, the running firmware **blacklists 2.0.40** locally to prevent a
re-download loop, so it will ignore any 2.0.40 image even if rebuilt correctly. The corrected build
therefore ships as **2.0.41**, a new version that the running firmware will accept and that is not
blacklisted.

## Deployment

After CI builds this release, assign `TankAlarm-Client-secure-v2.0.41.slot.bin` on Notehub to the
client device. The running firmware will accept it (newer, not blacklisted), MCUboot will validate
and swap it, and the new image will confirm itself on first boot.

> If 2.0.41 also reverts, the remaining suspect is **bootloader provisioning** — i.e. the device must
> have been flashed over USB with the genuine `security=sien` (MCUboot) bootloader for the core's
> keychain to be trusted. That would be a one-time USB re-flash, not a firmware change.
