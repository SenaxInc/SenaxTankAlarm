# v2.0.46 — A0602 shared‑bus hardening (I²C single‑owner discipline)

**Client‑only release.** Server/Viewer firmware is unchanged (the shared `TankAlarm-112025-Common`
version bump means they will report 2.0.46, but no server/viewer behaviour changed).

This release implements the firmware‑only plan from
`CODE REVIEW/OPTA_I2C_BUS_SEPARATION_FEASIBILITY_06242026.md` (§13, the consolidated plan of record) to make
the A0602 4‑20 mA current‑loop reads robust on the I²C bus they **share** with the Blues Notecard. No hardware
changes — the Opta exposes only one usable I²C bus, so the fix is bus *discipline*, not separation.

## What changed

### FT1 — Two‑phase sampling (the main fix)
`sampleMonitors()` now runs as two phases on devices that have a current‑loop monitor:
1. **Phase A** acquires + validates **every** sensor with **no** Notecard I/O.
2. **Phase B** (after all A0602 reads are done) evaluates alarms/unloads and publishes telemetry.

Previously `sendTelemetry()` (and alarm/unload publishes) ran **inside** the per‑monitor loop; each
`publishNote()` is a host‑initiated Notecard transaction that can hold the shared bus for up to ~30 s, landing
between two A0602 frames and corrupting the read. Two‑phase guarantees the A0602 owns the bus during its reads.
Devices with no current‑loop monitor keep the original interleaved behaviour.

### FT2 — Re‑balanced I²C timeouts
- **Notecard / global** `I2C_WIRE_TIMEOUT_MS`: **25 → 50 ms** (the Notecard clock‑stretches SCL up to ~25 ms
  during sync; 25 ms was marginal).
- **A0602** transactions: now use a **short scoped 25 ms** timeout (`A0602_WIRE_TIMEOUT_MS`). The A0602 does not
  clock‑stretch, so it fails fast into bus recovery instead of blocking (was a 100 ms scope).

### FT7 — Per‑op 400 kHz clock window
The A0602 framed transactions now run at **400 kHz**, scoped to the A0602 section only, with a **guaranteed
restore to 100 kHz before any Notecard op** on every return path. This shrinks the A0602's bus‑occupancy window
~4× **without** ever running the clock‑sensitive Notecard at 400 kHz. (`A0602_PEROP_I2C_CLOCK_HZ`,
`I2C_NORMAL_CLOCK_HZ`.)

### FT4 — Garbage / over‑range rejection
- `tankalarm_readCurrentAdcFramed()` now rejects a raw `0xFFFF` sample (it would otherwise scale to 25 mA).
- `readCurrentLoopSensor()` adds an **over‑range guard**: any averaged reading above `CURRENT_LOOP_OVER_RANGE_MA`
  (21 mA) is treated as a fault instead of being extrapolated by `linearMap()` into a fabricated level/pressure.

### FT3 — Recovery robustness
The existing post‑burst, backoff‑gated **sensor‑only I²C recovery** now re‑binds the Notecard
(`tankalarm_ensureNotecardBinding`) after `recoverI2CBus()` cycles the bus. The misleading `PB_8/PB_9` recovery
pin comment was corrected to the actual Opta `Wire` (I2C3) pins **PH_7 (SCL) / PH_8 (SDA)**.

### FT5 / FT6 — Observability (so Notehub is the validation surface)
New A0602 diagnostics, daily‑windowed (reset with the existing `i2c_cl_err`):
- `cl_ok` — valid A0602 reads in the window.
- `cl_or` — readings rejected as over‑range / `0xFFFF` garbage.
- `cl_fault` — last fault code (see table below).
- `cl_dur_us` — duration of the last A0602 read burst (confirms the 400 kHz window's effect).

These appear in the **daily health report**, the **daily I²C summary** note, and the **`i2c-recovery`
diagnostic** note (event‑driven).

#### `cl_fault` codes
| Code | Meaning |
|---|---|
| 0 | none (last read OK) |
| 1 | P1 power‑gate enable NACK (loop unpowered) |
| 2 | ADC channel config NACK |
| 3 | channel not in current‑ADC mode |
| 4 | every framed sample failed |
| 5 | reading rejected as over‑range / `0xFFFF` |

## Deferred (NOT in this release)
- **Global 50 kHz clock** — the *alternative* clock lever to the 400 kHz window. The two are mutually exclusive;
  the 400 kHz window was chosen because it cannot destabilise the Notecard. The 50 kHz option remains available if
  field data shows the bus is electrically marginal rather than temporally contended.
- **Physical verification (Phase 3)** — loop‑supply energisation on P1‑ON, pull‑ups, P1 wiring, A0602 rail
  inrush/brownout — is hardware work and cannot be delivered via OTA.

## Validation
Ship via OTA to the test client and watch Notehub: expect `cl_dur_us` to drop (400 kHz window working),
`i2c_cl_err` to fall and `cl_ok` to rise if contention was the cause, a real `ma`/`lvl` to appear, and `cl_fault`
to indicate the dominant remaining failure mode. If `i2c_cl_err` stays high with no valid `ma`, the cause is
physical (Phase 3).
