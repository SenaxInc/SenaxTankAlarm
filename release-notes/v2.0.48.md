# v2.0.48 ‚Äî Battery monitor sourcing rework + Voltage Monitor UI

**Client + Server release.** Viewer firmware is unchanged (the shared
`TankAlarm-112025-Common` version bump means it will report 2.0.48, but no Viewer behaviour changed).

This release implements the corrective fixes from
[CODE REVIEW/CODE_REVIEW_06252026_BATTERY_VOLTAGE_SAMPLING.md](../CODE%20REVIEW/CODE_REVIEW_06252026_BATTERY_VOLTAGE_SAMPLING.md):
the Notecard `card.voltage` API is permanently removed as a battery source, the
`batteryMonitor` infrastructure now consumes from the same source as the power-state machine
(SunSaver MPPT or analog Vin divider), and the server's Client Configuration page exposes a
single explicit Voltage Monitor dropdown instead of the legacy Vin Monitor toggle.

## What changed

### Fix 9 ‚Äî Strict source priority in `getEffectiveBatteryVoltage`
Voltage source selection is now strict priority instead of "lower of":

1. **Primary:** SunSaver MPPT via Modbus RS-485 (`"mppt"`) ‚Äî the digital charge-controller reading.
2. **Fallback:** Analog Vin voltage divider (`"vin-divider"`) ‚Äî only when MPPT is unavailable.

Previously, an enabled Vin divider that read lower than a healthy MPPT could override the MPPT
value via the conservative `min()` selection. A noisy/uncalibrated divider could therefore drag
a clean 12.3 V MPPT reading down to e.g. 11.9 V and trip a bogus `CRITICAL_HIBERNATE`. With
strict priority the MPPT, when healthy, is never overridden.

### Fix 10 ‚Üí Fix 11 ‚Äî Notecard `card.voltage` removed as a battery source on all platforms
Fix 10 originally compile-gated the Notecard path out on Opta/mbed. Fix 11 takes this further
and removes the Notecard battery path entirely on **every** build:

- **`readNotecardVinVoltage()` deleted** ‚Äî dead code; never called after Fix 7.
- **`configureBatteryMonitoring()` deleted** ‚Äî used to push voltage thresholds into the
  Notecard's `card.voltage` mode string; that API path is no longer used.
- **`pollBatteryVoltage()` refactored** ‚Äî now reads `getEffectiveBatteryVoltage()` and the
  `gEffectiveVoltageSource` global instead of issuing a Notecard `card.voltage` request. The
  function is now hardware-agnostic; no platform compile guard remains around its call sites
  in setup, loop, or the daily report.
- **`BatteryData.mode` repurposed** ‚Äî was the Notecard voltage-mode classifier
  (`"usb"` / `"high"` / `"normal"` / `"low"` / `"dead"`); now holds the source identifier
  (`"mppt"` / `"vin-divider"`). Other Notecard-specific fields
  (`usbPowered`, `dailyChange`, `weeklyChange`, `monthlyChange`, `isCharging`, `isDeclining`,
  `voltageAvg`, `analysisHours`) are marked reserved and remain populated with safe defaults.

Rationale: on the Blues "Wireless for Opta" carrier the Notecard's V+ is supplied through the
carrier's regulated ~5 V DC-DC converter and physically cannot represent the 12 V battery.
Off-Opta hardware was the only place the old Notecard path could have read true battery
voltage, and there is no remaining deployment that relies on that path ‚Äî so the simpler unified
sourcing model is now used everywhere.

### Fix 11 ‚Äî De-duplicated alert pipeline
`checkBatteryAlerts()` returns early when `data.mode == "mppt"`. The SunSaver manager already
publishes its own low/critical/high alerts via `solarCharger.alertOnLowBattery`, so this avoids
sending duplicate `y: "battery"` notes for the same MPPT reading. The `batteryMonitor` alert
pipeline retains ownership of alerts when the active source is `"vin-divider"`.

### Daily-report source tagging
`appendBatteryDataToDaily()` now emits a new `battery.src` field carrying the active source
name. The existing telemetry-level `vs` tag (Fix 7) is preserved.

### Server UI ‚Äî new Voltage Monitor dropdown
The Client Configuration page (`/config-generator`) replaces the two-option **Vin Monitor**
selector with a three-option **Voltage Monitor** dropdown:

| Option | Effect on config JSON |
|---|---|
| **None** | `vinMonitor.enabled = false`, `solarCharger.enabled = false` |
| **Voltage Divider** | `vinMonitor.enabled = true`, `solarCharger.enabled = false` |
| **RS485 (SunSaver Modbus)** | `vinMonitor.enabled = false`, `solarCharger.enabled = true` |

The new selector is the source of truth for `solarCharger.enabled` (it is no longer derived
from `powerSource === 'solar_modbus_mppt'`). Legacy saved configs are round-tripped:
`solarCharger.enabled` ‚áí `"rs485"`, `vinMonitor.enabled` ‚áí `"vin-divider"`. A new top-level
`voltageMonitor` field is emitted in the config JSON for the client's benefit; older firmware
will safely ignore it.

A non-blocking validation warning now fires when `powerSource === 'solar_modbus_mppt'` but
`voltageMonitor !== 'rs485'`, reminding the operator that a SunSaver-Modbus install needs RS485
voltage monitoring to actually read the controller.

## What is intentionally NOT in this release

- **Server-side dashboard surfacing of the active source** ‚Äî `ClientMetadata` already stores
  the voltage via `vinVoltage`/`vinVoltageEpoch` from the top-level `v`/`vs` fields, but does
  not persist the source string. A future patch can add `char vinSource[16]` to expose the
  source name on the dashboard / API.
- **Viewer source-tag pass-through** ‚Äî the Viewer ingests `item["v"]` but does not echo
  `item["vs"]` through its API. A future patch can add `batterySource` to the Viewer's record.
- **Multi-day trend statistics** ‚Äî the Notecard provided 24h/7d/30d voltage trends from its
  own historical window. After removing the Notecard source, firmware tracks only per-boot
  running min/max. The `dailyChange` / `weeklyChange` / `monthlyChange` fields remain in the
  struct for binary compatibility but are always 0; emitters skip them.

## Backward compatibility

- **Server config schema**: the `vinMonitor.enabled` / `solarCharger.enabled` booleans are
  still emitted by the server; the new `voltageMonitor` enum is additive. Older firmware
  ignores it.
- **Legacy `card.voltage` readings on the server**: existing `ClientMetadata.vinVoltage`
  entries below 6 V (the Notecard 5 V rail leak) are already filtered by the server's
  plausibility gate (Fix 6/7, pre-existing).
- **Notefile schema**: unchanged. `NOTEFILE_SCHEMA_VERSION` stays at 1.
