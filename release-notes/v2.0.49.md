# v2.0.49 — RS-485 / SunSaver MPPT Modbus hardening

**Client-only release.** Server and Viewer firmware are unchanged (the shared
`TankAlarm-112025-Common` version bump means they will report 2.0.49, but no server/viewer
behaviour changed).

This release implements the corrective fixes from
[CODE REVIEW/CODE_REVIEW_06252026_RS485_SUNSAVER_COMMUNICATION.md](../CODE%20REVIEW/CODE_REVIEW_06252026_RS485_SUNSAVER_COMMUNICATION.md):
the RS-485 DE turnaround now scales with the configured baud rate, the Modbus error
counter is reset per daily-report window, the realtime poll no longer starves the
cooperative pulse sampler when the bus is dead, and the production `chargeState` no longer
permanently reports `"Starting"`.

## What changed

### Fix S1 — Dynamic RS-485 DE turnaround delay
`RS485.setDelays(0, 1200)` was a static upper bound that worked at the default 9600 baud
(11-bit character time = 1146 µs) but at the higher baud rates the sanitizer permits
(19200, 38400, 57600, 115200) the 1200 µs hold was longer than the slave's 3.5-character
inter-frame gap, causing **bus contention** between the Opta DE driver and the slave's
response. The post-TX delay is now computed dynamically:

```cpp
uint32_t postDelayUs = 11000000UL / _config.modbusBaudRate;
if (postDelayUs < 150)  postDelayUs = 150;   // floor (protects 115200 from DE underrun)
if (postDelayUs > 1500) postDelayUs = 1500;  // ceiling
RS485.setDelays(0, postDelayUs);
```

Behavior at each supported baud:

| Baud | Computed delay | Was | Status |
|---:|---:|---:|---|
| 9600 | 1145 µs | 1200 µs | unchanged (default) |
| 19200 | 573 µs | 1200 µs | no longer crowds slave response |
| 38400 | 286 µs | 1200 µs | no longer crowds slave response |
| 57600 | 191 µs | 1200 µs | no longer crowds slave response |
| 115200 | 150 µs (floor) | 1200 µs | no longer crowds slave response |

The boot log now prints the computed `postDelay` value alongside baud, parity, slave ID,
and timeout for easier field verification. Production deployments running at the default
9600 baud see no behaviour change.

### Fix S2 — Modbus error counter reset per daily window
`SolarManager::resetModbusErrorStats()` had no callers; the `merr` field in daily reports
tracked **lifetime-since-boot** error counts, which made it impossible to tell whether the
bus was currently unstable or had simply accumulated old transients.

Added `gSolarManager.resetModbusErrorStats();` next to the existing
`gCurrentLoopI2cErrors = 0;` reset block at the end of `sendDailyReport()`. `merr` is now a
24-hour window count and matches the I2C error reporting pattern.

### Fix S3 — Adaptive realtime poll: no more 6-second loop starvation on dead bus
Previously, when the SunSaver bus was down, each failed register read tried both FC03 and
FC04 (1 s timeout each = 2 s per read), and the realtime block retried up to 3 times with
100 ms between attempts → **worst case ~6.2 sec of synchronous blocking per poll**. That
starved the cooperative `pollPulseSampler()` state machine, causing missed edges on engine
RPM and flow sensors.

The fix is a `fastFail` flag plumbed from `readRegisters()` down through
`readRegistersIndividually()` and `readRegistersWithFallback()`:

```cpp
const bool fastFail = (_data.consecutiveErrors >= 1);
const uint8_t maxAttempts = fastFail ? 1 : SOLAR_REALTIME_MAX_ATTEMPTS;
```

When `fastFail` is true:
- The realtime retry loop caps at **1 attempt** (no retries).
- Each register read **skips its alternate-FC probe** (the FC cache is still preserved, so R-4 transient-tolerance hardening from v2.0.45 is intact).

Worst-case dead-bus polling cost drops from **~6.2 s** to **~1 s** as soon as the second
consecutive failure starts. Pulse sampler resumes between polls instead of after the full
comm-failure-threshold window. The first failure still pays the full 6.2 s cost so a
single transient is still absorbed by alternate-FC probing.

### Fix S4 — Derived `chargeState` in production
The production `#else` branch in `readRegisters()` hardcoded
`nextData.chargeState = CHARGE_STATE_START`, which made
`getChargeStateDescription()` permanently report `"Starting"` even when the charger had
been in Float mode for months.

Added a software-derived `chargeState` in `updateHealthStatus()` based on the verified
electrical readings:

| Condition | Derived state |
|---|---|
| `isFullyCharged` (not charging + battery at float voltage) | `CHARGE_STATE_FLOAT` |
| `isCharging` (current flowing in) | `CHARGE_STATE_BULK` |
| `arrayVoltage < batteryVoltage + 1.0 V` | `CHARGE_STATE_NIGHT` |
| Otherwise | `CHARGE_STATE_DISCONNECT` |

`ABSORPTION` and `EQUALIZE` both fall under `BULK` because distinguishing them requires
verified status-register addresses. See the bench-verification TODO in
`TankAlarm_Solar.cpp` for the path to richer state reporting.

### Inconsistency E — documentation only
The hardware fault / alarm registers at `0x002B`, `0x002C`, `0x002E` returned
`faults=0x4235` on a physically healthy unit during 2026-04-22 bench-up; activation
without further verification would cause false-alarm SMS spam to every field unit. The
existing `SOLAR_ENABLE_UNVERIFIED_REGISTERS` guard stays in place; this release adds:

- A `BENCH-VERIFICATION TODO` block to the existing field-ready-policy comment in
  `TankAlarm_Solar.cpp` listing the specific addresses to probe (`0x0021`, `0x0023`,
  `0x002F`, `0x0032`), the conditions to induce on the bench (DIP flip, RTS disconnect,
  array short), and the staged-rollout path for eventual activation.
- A "Known gaps in fault detection" subsection in the client README's Solar Charger
  section. It enumerates which alerts ARE produced from verified electrical readings
  (battery low/critical/high, no-charge, comm failure) and which failure modes are NOT
  currently surfaced (DIP tampering, EEPROM corruption, RTS wiring, battery sense wiring,
  calibration drift, heatsink overtemp), with cross-references to the code review and the
  TODO comment.

## What is intentionally NOT in this release

- **Activation of the fault/alarm register reads.** Requires bench verification that has
  not yet been completed; activating with wrong addresses would emit false-alarm SMS to
  every field install. See the new TODO comment in `TankAlarm_Solar.cpp` and the
  "Known gaps" subsection in the client README for the path to enable.

## Backward compatibility

- **`readRegistersWithFallback` / `readRegistersIndividually` signatures**: the new
  `fastFail` parameter defaults to `false`, so every existing caller (setpoint block,
  unverified-register blocks) keeps its current behavior. Only `readRegisters()` opts in.
- **Server config schema**: unchanged. `solarCharger` block fields are exactly the same.
- **Notefile schema**: unchanged. `NOTEFILE_SCHEMA_VERSION` stays at 1.
- **`merr` field semantics in daily reports**: was lifetime-since-boot, now 24-hour
  windowed. Server-side consumers should not break (it's still a non-negative integer);
  dashboards interpreting `merr` as cumulative may want to note the meaning change.
