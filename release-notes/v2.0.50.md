# v2.0.50 — Current-loop sensor hardening (Fix C1 + Fix C2)

**Client-only release.** Server and Viewer firmware are unchanged (the shared
`TankAlarm-112025-Common` version bump means they will report 2.0.50, but no server/viewer
behaviour changed).

This release implements the corrective fixes from
[CODE REVIEW/CODE_REVIEW_06252026_CURRENT_LOOP_SENSORS.md](../CODE%20REVIEW/CODE_REVIEW_06252026_CURRENT_LOOP_SENSORS.md)
§10 ("Best-Path Implementation Specification"), after five review passes converged on a
narrow, surgical patch.

## What changed

### Fix C1 — Lightweight post-recovery A0602 address re-probe
After an in-loop I2C bus recovery (`recoverI2CBus()` → SCL toggle → `Wire.end()/begin()`),
the A0602 normally retains its Blueprint-assigned address (its MCU keeps running on its own
12–24 V supply). But if the module was externally power-cycled — brownout, loop-power loss,
hardware reset — it lost its `0x0B` assignment and is now listening at the
`OPTA_DEFAULT_SLAVE_I2C_ADDRESS` (`0x0A` == `CURRENT_LOOP_I2C_ALT_ADDRESS_1`). The previous
firmware kept polling at the now-stale `0x0B` and never recovered until reboot.

After C1, both in-loop bus-recovery sites now:
1. Wait 50 ms for the bus peripheral to settle.
2. Call `resolveCurrentLoopI2cAddress(prev)` (3 lightweight `i2cAck` pings — `{preferred, 0x64, 0x0A}`).
3. If a different address ACKs, update `gConfig.currentLoopI2cAddress` **at runtime only** (NOT persisted, NOT marked dirty).
4. If the re-probe lands on `0x0A`, emit a `WARNING` log line distinguishing "address found" from "sensor working" — see §10 notes.

**Important design constraint that was rejected during review:** the original Inconsistency B
recommendation was to call `bootstrapA0602Managed()` from the recovery wrapper. That was
rejected because:
- The function's own comment at `bootstrapA0602Managed()` explicitly forbids calling it outside `setup()`.
- It emits `OPTA_CONTROLLER_RESET` to every Blueprint expansion (broadcast disruption).
- It forces the bus to 400 kHz internally (Notecard collision risk).
- Cost ≥ 500 ms per call — would starve the cooperative pulse sampler when recoveries are frequent.

The C1 re-probe is the safer alternative: ≤150 ms worst case, no Blueprint broadcast, no
clock change, no prohibition violation.

**Surgical placement:** C1 is added at the two sensor-recovery call sites only
(L1987-L2019 dual-failure, L2071-L2097 sensor-only), NOT inside the `recoverI2CBus()`
wrapper. This avoids the 50–200 ms latency hit on the two Notecard-only recovery sites
inside `checkNotecardHealth()` where the A0602 bus is healthy.

### Fix C2 — Configure ADC channel BEFORE PWM warmup; split burst windows
The previous flow was: PWM-enable → 3 s warmup → open 400 kHz burst → configure ADC →
priming read → sample → close burst → PWM-disable. During the 3 s warmup the A0602
channel sat in default/high-impedance mode, so the AD74412R sense node was not connected
to the I/O pin. The transmitter had no closed loop circuit to drive current into during
its electrical stabilization window — the first priming read therefore captured a
freshly-configured sense node that hadn't completed its own settling.

After C2, the flow is split into two burst windows:

1. **PWM enable** at normal 100 kHz bus speed (no longer inside a high-speed window).
2. **CONFIGURE burst window**: open 400 kHz / 25 ms timeout → configure ADC + `GET_CHANNEL_FUNCTION` verify (existing 3-retry, unchanged) → close (back to 100 kHz / normal timeout).
3. **Warmup delay** (chunked 1 s with WDT kicks) — **now with the ADC channel already configured**, so the AD74412R sense resistor is connected to the I/O pin and loop current can stabilize electrically during the entire warmup window.
4. **SAMPLE burst window**: re-open 400 kHz → priming read + N samples → close (back to 100 kHz).
5. **PWM disable** at normal 100 kHz bus speed.

**Bonus**: a failed configure now bails BEFORE the 3 s warmup, saving ~3 s per
failed-config read attempt.

**Critical implementation constraints** (each one is a numbered finding from the review):
- All five existing side-effects are preserved on every early-return path: `gCurrentLoopI2cErrors++`, `gLastClFaultReason = ...`, `gMonitorState[idx].currentSensorMa = 0.0f`, `gMonitorState[idx].sampleReused = true`, and a proper 3-retry PWM-off at 100 kHz (§9-1). The PWM-off in error paths used to be a single-shot `(void)tankalarm_setPwm(...)`; it now uses the same retry loop as the normal-exit path.
- `gLastClBurstMicros` uses a two-delta accumulator: `configWindowUs + sampleWindowUs`, excluding the warmup gap (§9-2). The `cl_dur_us` telemetry field therefore still represents "active A0602 bus time" and historical dashboards stay comparable.
- The Wire window is **always closed before any cleanup commands** (PWM-off, warmup delay) so the 400 kHz / short-timeout settings cannot leak to a Notecard transaction (§8.2.1, §8.2.2).
- The `tankalarm_configureCurrentAdcChannel()` doc-comment in [TankAlarm_I2C.h](../TankAlarm-112025-Common/src/TankAlarm_I2C.h#L397) is updated to reflect the new "configure BEFORE warmup" contract (§8.2.4).

## Status of the field "stuck ~18 mA" symptom

C1 + C2 are **hardening, not the cure** for the production "stuck ~18 mA / 43.8 psi" symptom.
The cure is the existing setup-time `bootstrapA0602Managed()` handshake (v2.0.47).

Bench validation status as of 2026-06-25:
- **Confirmed**: managed bootstrap drives the A0602 status LED to **GREEN** on this hardware.
- **Pending**: end-to-end "raw framed read returns a sane ~4 mA at no-fluid baseline" capture.

Until the mA trace is recorded, the §5 "Address Targeting" and "Gated Stabilization"
checklist rows in the review document remain **PROVISIONAL** rather than VERIFIED.

If post-deployment telemetry shows the C1 WARNING line firing in production (a recovery
landing on `0x0A`), the next priority is the deferred maintenance-window re-management
path (turn an unmanaged module back to GREEN at 0x0B without violating the bootstrap
prohibition).

## What is intentionally NOT in this release

- **`bootstrapA0602Managed()` from `recoverI2CBus()`** — violates the explicit setup-only prohibition; replaced by Fix C1.
- **Upper clamp for the 20–21 mA "fault-high" zone** — existing 21 mA `CURRENT_LOOP_OVER_RANGE_MA` cutoff matches NAMUR NE-43; the original Inconsistency D wording was advice, not a code change.
- **`y:"sensor-degraded"` telemetry note** when C1 re-probe lands on `0x0A` — for now the WARNING log line is the operator-visible signal. A formal telemetry event can be added once the field shows whether unmanaged modules service framed reads (see §10.5 row 4).
- **True A0602 re-management after external reset** (RED→GREEN, 0x0A→0x0B reassignment) — requires a maintenance-window bootstrap path or watchdog-triggered soft Opta reboot. Deferred to a future review.
- **Server-side dashboard surfacing** of the new WARNING / re-probe events — out of scope for the firmware patch.

## Backward compatibility

- **Notefile schema**: unchanged. `NOTEFILE_SCHEMA_VERSION` stays at 1.
- **Server config schema**: unchanged. No fields added or removed.
- **`gLastClBurstMicros` / `cl_dur_us` semantics**: deliberately unchanged. The two-delta
  accumulator excludes the warmup gap so historical trend continuity is preserved.
- **`gA0602ManagedAddress`**: still set only during `setup()`; the C1 re-probe does NOT
  update it. If the re-probe finds the module at a non-managed address, that global will
  reflect the original setup-time assignment (effectively stale). It is currently used only
  for boot-time diagnostics, so this is benign.
