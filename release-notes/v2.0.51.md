# v2.0.51 — Current-loop telemetry: raw mA only, descriptive fault on failure (Fix 12)

**Client-only release.** Server and Viewer firmware are unchanged.

This release responds to a diagnostic gap observed on the first post-v2.0.50 telemetry:
the Cox Wellhead current-loop sample reported `lvl: 0` with `ru: 1` and no `ma` field,
which is ambiguous between a real live-zero reading and a failed read whose stale default
value got reused. The fix makes current-loop telemetry self-describing so a faulted read
can never look like a successful 0 PSI / 0 inches reading.

## What changed — Fix 12 (current-loop telemetry payload)

`buildSensorObject()` in [TankAlarm-112025-Client-BluesOpta/TankAlarm-112025-Client-BluesOpta.ino](../TankAlarm-112025-Client-BluesOpta/TankAlarm-112025-Client-BluesOpta.ino#L6276)
is the single helper that serializes sensor data into all three outbound notes
(`telemetry.qo`, the per-monitor array in `daily.qo`, and `alarm.qo`). For `SENSOR_CURRENT_LOOP`
sensors only:

| Path | Before (v2.0.50) | After (v2.0.51) |
|---|---|---|
| Successful read | `ma` only if ≥ 4.0 mA; `lvl` always emitted | `ma` always emitted; `lvl` / `cap` NOT emitted |
| Failed read (sampleReused) | `lvl: 0` (looks like a real 0); `ma` omitted; `ru: 1` flag | `fault: "<short_string>"` emitted; `ma` / `lvl` / `cap` NOT emitted; `ru: 1` flag kept |

The new `fault` field is one of these stable short strings (mapped from the existing
`gLastClFaultReason`):

| Value | Source path |
|---|---|
| `"pwm_nack"` | PWM gate-enable NACKed (transmitter loop unpowered) |
| `"config_nack"` | A0602 ADC channel configure NACKed every retry |
| `"func_wrong"` | `GET_CHANNEL_FUNCTION` returned a non-current-ADC mode |
| `"read_fail"` | Every framed sample NACK'd / CRC-failed |
| `"over_range"` | Reading exceeded `CURRENT_LOOP_OVER_RANGE_MA` (21 mA / 0xFFFF garbage) |
| `"ok"` / `"unknown"` | Last fault was none, or enum was out of range (should not normally appear) |

Other sensor types (digital float, analog voltage, pulse RPM) are **unchanged** — they
still emit their existing `fl`/`vt`/`rm` raw fields plus `lvl` + `cap` for backward
compatibility with the existing dashboard.

## Why

In v2.0.50 the first post-flash telemetry payload looked like this:

```json
{ "fv": "2.0.50", "st": "currentLoop", "lvl": 0, "pg": 1, "ru": 1, ... }
```

This is ambiguous: `lvl: 0` could mean
- the wellhead is genuinely at 0 PSI,
- the transmitter is at live-zero (~4 mA mapped through linearMap to 0),
- **or** the A0602 read failed and the firmware reused the default-zero `currentInches` value.

The `ru: 1` flag distinguishes the third case from the first two, but the missing `ma`
field still doesn't tell the operator (or the dashboard) *which* fault path tripped.
After Fix 12 the same scenario serializes to:

```json
{ "fv": "2.0.51", "st": "currentLoop", "pg": 1, "ru": 1, "fault": "config_nack" }
```

Now the operator can see immediately that the failure is in the A0602 channel-configure
step, not in the framed read, not in the PWM gate, and not in the over-range branch.

## Trade-offs and design notes

- **Server-side dashboard impact.** The existing server / dashboard renders `lvl` for the
  level value shown on the sensor card. After Fix 12, current-loop sensors will NOT carry
  `lvl` in any of the three note types (telemetry, daily, alarm). The server already has
  the sensor's `sensorRangeMin`/`sensorRangeMax`/`mu` from the config it generated; the
  natural follow-up is to teach the server to convert `ma` → engineering units itself,
  so the dashboard can show both raw mA and the derived level (or the `fault` string)
  with one consistent rendering path. **This server work is not in this release.** In the
  interim, dashboards that read `lvl` will show no value for current-loop sensors and
  should fall back to displaying `ma` (when present) or `fault` (when present).
- **Client-side alarm logic is unaffected.** The client still evaluates threshold alarms
  against `state.currentInches` locally; Fix 12 only changes what the outbound payload
  carries. Alarm notes (`alarm.qo`) still include `y:"high"`/`"low"`/etc., the relay
  control still fires, and SMS escalation still works.
- **Daily-report continuity.** Per-monitor entries in the `daily.qo` payload now use the
  same new layout; historical aggregates that rely on `lvl` for current-loop monitors
  will see no value after Fix 12. If a dashboard needs back-compat with old daily
  payloads, treat a missing `lvl` + present `ma` as the new format and convert.
- **Other sensor types untouched.** Digital float switches still emit `fl` + `lvl`,
  analog still emits `vt` + `lvl`, pulse still emits `rm` + `lvl`. Only current-loop
  gets the raw-only treatment.
- **`gLastClFaultReason` is a global**, not per-monitor. With a single current-loop
  monitor (the typical deployment) this is fine. With multiple current-loop monitors,
  the `fault` string emitted in a failed sample reflects the **last** fault seen across
  all monitors. A per-monitor cache is a small follow-up if multi-current-loop deployments
  become common.

## Backward compatibility

- **Notefile schema**: unchanged. `NOTEFILE_SCHEMA_VERSION` stays at 1.
- **Server config schema**: unchanged.
- **`lvl` field semantics**: when present, still means "client-computed engineering-units
  value." Now ABSENT on current-loop notes — server should detect this and either render
  `ma` directly or do its own conversion.
- **Existing `ru: 1` flag**: kept on current-loop fault paths in addition to the new
  `fault` string, so dashboards reading either signal continue to work.
- **No change to alarm-firing logic, relay control, SMS escalation, or sample cadence.**

## What to look for after flashing

1. **Confirm post-flash telemetry shape** — first sample after boot should be either
   `{... "ma": 4.xx, "pg": 1}` (healthy read) or `{... "pg": 1, "ru": 1, "fault": "config_nack" }`
   (or similar fault string).
2. **If `fault` appears, capture USB serial output** to cross-check the matching
   `ERROR: A0602 ...` line for the same poll, and confirm the enum → string mapping.
3. **Watch for the same fault repeating** across consecutive samples — that means the
   root-cause investigation should focus on the named code path.
4. **Daily report (`daily.qo`)** at 05:00 should also use the new layout for current-loop
   monitors.
