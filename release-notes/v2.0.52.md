# v2.0.52 — Server-side mA→level conversion + dashboard fault badge + raw mA display (Fix 13)

**Server-only release.** Client and Viewer firmware are unchanged. Existing
v2.0.51 clients in the field benefit immediately — no client reflash required.

This release closes the dashboard-visibility gap opened by v2.0.51 (Fix 12).
After Fix 12 the client stopped sending `lvl` for current-loop sensors and started
sending raw `ma` on success or a `fault` string on failure; the server still
required a `lvl` value to display anything, so current-loop sensors showed nothing
or stale-zero on the live dashboard.

## What changed — Fix 13

### 1. Server-side mA → engineering-units conversion

[`resolveLevel()`](../TankAlarm-112025-Server-BluesOpta/TankAlarm-112025-Server-BluesOpta.ino) now
derives `levelInches` server-side when a current-loop sensor sends valid `ma`
without a `lvl` field. It reuses the existing
[`convertMaToLevelWithTemp()`](../TankAlarm-112025-Server-BluesOpta/TankAlarm-112025-Server-BluesOpta.ino)
helper, so server and client share identical math:

- **Learned calibration available** → apply learned slope/offset + live temperature compensation
- **No learned calibration** → theoretical conversion from the cached client config
  (`sensorRangeMin`, `sensorRangeMax`, `sensorMountHeight`, `currentLoopType`,
  `sensorRangeUnit`, `fluidSpecificGravity` / `fluidType`)

This branch only runs when the client omitted `lvl` — older clients (≤ v2.0.50)
that still send `lvl` for current-loop are unaffected.

### 2. Server `SensorRecord.sensorFault` — transient fault state

A new `char sensorFault[16]` field on
[`SensorRecord`](../TankAlarm-112025-Server-BluesOpta/TankAlarm-112025-Server-BluesOpta.ino)
holds the latest current-loop fault string (`"pwm_nack"`, `"config_nack"`,
`"func_wrong"`, `"read_fail"`, `"over_range"`).

- Always cleared on every incoming telemetry / alarm note, then set if the note carries `"fault": "<reason>"`.
- Reflects the **latest** read's health — never historical.
- Not persisted to the sensor registry (transient: refreshed by the next telemetry).

### 3. Dropped the `>= 4.0 mA` gate on stored / emitted `ma`

Four sites previously gated `obj["ma"]` at `>= 4.0f`:
- `sendSensorJson()` (`/api/sensors`)
- `sendClientDataJson()` (`/api/clients`) — both client-level and per-sensor sites
- `saveSensorRegistry()` (LittleFS persistence)

After Fix 13 these all emit when `sensorMa > 0.0f`. The client (post-Fix-12) now
sends `ma` only on a successful read — there is no longer a reason to silently
drop a real low-but-valid reading.

### 4. API gains `flt` field

All three API serialization sites that already emit `ma` now also emit
`flt = sensorFault` when non-empty. The new shape for a faulted current-loop
sensor returned by `/api/clients?summary=1`:

```json
{ "c": "dev:xxx", "n": "Cox Wellhead", "st": "currentLoop",
  "l": 0, "flt": "config_nack", "u": 1735000000 }
```

And for a healthy current-loop sensor:

```json
{ "c": "dev:xxx", "n": "Cox Wellhead", "st": "currentLoop",
  "l": 42.7, "ma": 14.65, "u": 1735000000 }
```

### 5. Dashboard renders fault badge and raw mA

The dashboard cards (`renderDataCard`) and the per-client expanded sensor rows
(`tanksHtml`) now branch on the new `t.fault` field:

- **Fault present** → big red `FAULT` in place of the level value, with the
  fault reason in small red text next to it. The dashboard no longer shows
  a misleading `0` value when the sensor is in a fault state.
- **Healthy current-loop reading** → big converted value (e.g. `42.7 in`) with
  the raw mA value (e.g. `14.65 mA`) in small gray text directly below. The
  operator sees both the engineering-units value *and* the raw transmitter
  current, which together immediately disambiguate "tank at 0 PSI" from
  "transmitter at live-zero" from "wire fault".
- **Other sensor types** (digital float, analog voltage, pulse) — unchanged.

The model mapper in `buildSiteModel()` propagates `t.flt` from the API payload
to `sensor.fault` so the JS render code can rely on it.

## Why

After v2.0.51 the first telemetry from Cox Wellhead arrived with:

```json
{ "fv": "2.0.51", "st": "currentLoop", "fault": "config_nack", "ru": 1 }
```

This was exactly what Fix 12 was meant to deliver — a clean self-describing
fault. But the dashboard had no UI for it: the `lvl` field was absent, so the
sensor card showed a stale zero or nothing. Fix 13 closes the loop end-to-end.

## Trade-offs and design notes

- **Backward compatible with all client versions.** v2.0.50 and earlier still
  send `lvl` for current-loop → server trusts it (existing path); v2.0.51+
  send raw `ma` + optional `fault` → server derives `lvl` and surfaces fault.
- **Forward compatible with future clients.** A v2.0.52+ client sending both
  `ma` and `fault` (defensive) would have `fault` take priority in the UI.
- **`sensorFault` is per-sensor, not per-client.** A multi-current-loop deployment
  with one faulted and one healthy sensor will correctly show one fault badge
  and one healthy card. The existing `gLastClFaultReason` global on the *client*
  is still the source of the `fault` string, but the server's `SensorRecord`
  storage is keyed by `(clientUid, sensorIndex)` so per-sensor state is preserved
  on the server side regardless.
- **Live-zero readings now visible.** A sensor reading 3.6 mA (~live-zero, below
  the 4-20 mA nominal range) now appears on the dashboard as `0 in` with
  `3.60 mA` underneath. Previously the `>=4.0` gate silently hid this. The
  operator can now distinguish "transmitter alive but reporting near-empty"
  from "transmitter completely unpowered" (the latter would emit a `fault`).
- **No client reflash required** for existing v2.0.51 deployments. Server
  upgrade alone delivers the full UI improvement.
- **No new persistence keys.** `sensorFault` is transient — not saved in the
  LittleFS sensor registry. On server reboot, the field starts empty and is
  refreshed by the next telemetry note.

## Backward compatibility

- **API consumers**: a new optional `flt` string field may appear on sensor
  objects. Old consumers that ignore unknown fields are unaffected.
- **Notefile schema**: unchanged. `NOTEFILE_SCHEMA_VERSION` stays at 1.
- **Sensor registry on disk**: unchanged shape. `sensorFault` is not persisted.
- **Dashboard JS**: defaults are safe — `t.fault || ''` makes missing fields a no-op.
