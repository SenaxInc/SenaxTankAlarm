# v2.0.53 — RS-485 / SunSaver observability (Fix R1–R6) + server dashboard readout

**Client + Server release.** Common firmware bumped to 2.0.53 / build seq 247. Both
the client and the server need to be flashed for the full observability benefit;
running a v2.0.53 server with v2.0.51/52 clients is forward-compatible (new
fields are simply absent for older clients).

This release implements the six findings from [CODE REVIEW/CODE_REVIEW_06252026_RS485_TELEMETRY_FOLLOWUP.md](../CODE%20REVIEW/CODE_REVIEW_06252026_RS485_TELEMETRY_FOLLOWUP.md). Telemetry payloads grow ~12 B happy-path / ~62 B failed-path. A complementary server-side dashboard readout (Fix 14) renders the new fields on both the main dashboard (per-client status dot expansion) and the site-config page (RS-485 badge in each client header).

## What changed — Fix R1–R6 (client + common)

### R1 — Per-poll error type, response time, and failing register address

`captureSolarModbusError()` now also classifies the library `lastError()` string into one
of five short tags: `"to"` (Timeout), `"crc"` (CRC error), `"ida"` (Illegal data address),
`"ifu"` (Illegal function), or `"?"` (other). Stored in a file-static
`sSolarLastErrorTag[8]` and exposed via the new `SolarManager::getLastModbusErrorTag()`
accessor.

Telemetry adds (only when `scOk:0`):
```jsonc
"scErr":   "to|crc|ida|ifu|?",
"scResMs": <uint16>,   // last response ms
"scMaddr": <uint16>    // last failing register address
```

### R2 — First / last successful read timestamps

`SolarData` now tracks `firstReadMillis` (millis() of the very first successful read since
boot, 0 = never). The existing `lastReadMillis` already tracked last success.

Telemetry adds:
```jsonc
"scOkEver":  0 | 1,        // 1 = link has succeeded at least once since boot
"scLastOkS": <uint32>      // seconds since last successful read (only if non-zero)
```

This lets the server distinguish *"never worked since this reboot"* (likely wiring /
baud / slave mismatch) from *"was working then degraded N seconds ago"* (more likely
power glitch or noise event).

### R3 — Decoupled the "library init" state from the emit guard

Before: `if (gSolarManager.isEnabled())` — which is `_config.enabled && _initialized` —
gated the entire telemetry/daily solar block. A failed `ModbusRTUClient.begin()` at boot
made the client emit nothing, indistinguishable from "user disabled solar in config".

After: `if (gConfig.solarCharger.enabled)` is the emit guard, and a new
`SolarManager::isInitialized()` accessor lets the client emit `scInit:0` when the library
failed to init.

Telemetry adds (always when solar is configured):
```jsonc
"scInit": 0 | 1
```

When `scInit:0`, the rest of the solar fields are omitted (no library = no diagnostics).

### R4 — Setpoint readback as a "Modbus link is real" signal

`SolarData::setpointsValid` becomes true when the chemistry-setpoint registers
(`V_reg`, `V_float`, `V_eq`) are read successfully at startup. This is the strongest
possible confirmation that the SunSaver and the Opta are speaking Modbus correctly.

Telemetry adds (always when solar is configured and `scInit:1`):
```jsonc
"scSpv": 0 | 1
```

A `scOk:0 scSpv:1` combination means the link decoded at least once since boot —
likely a transient runtime issue. A `scOk:0 scSpv:0` combination means the link has
never decoded — likely a configuration or hardware problem.

### R5 — Per-error-type bucket counters in the daily report

Five new file-static counters (`sSolarErrTimeout`, `sSolarErrCrc`, `sSolarErrIllegalAddr`,
`sSolarErrIllegalFunc`, `sSolarErrOther`) are incremented in
`readRegistersWithFallback()` alongside the existing `sSolarModbusErrorCount`. In steady
state the five buckets sum to the lifetime total. Reset together by
`resetModbusErrorStats()` (now called from `sendDailyReport()` per Fix S2 from v2.0.49).

Daily report (`appendSolarDataToDaily`) adds, in both success and failure branches:
```jsonc
"merrTo":  <uint32>,   // Timeout count this daily window
"merrCrc": <uint32>,   // CRC error count
"merrIda": <uint32>,   // Illegal data address count
"merrIfu": <uint32>,   // Illegal function count
"merrOth": <uint32>    // Other / catch-all
```

On the success path, only non-zero buckets are emitted (so a clean day emits nothing
extra). On the failure path, all five are always emitted.

### R6 — Single-poll truth alongside the 5-poll smoothed `scOk`

`SolarData::lastPollOk` mirrors the success of the most recent poll without smoothing.
Set every poll in `readRegisters()`. The existing `communicationOk` still uses the
5-poll `SOLAR_COMM_FAILURE_THRESHOLD` smoothing for SMS escalation — `scOk` continues
to drive SMS alerts; `scLast` is the dashboard-honest indicator.

Telemetry adds:
```jsonc
"scLast": 0 | 1     // single-poll truth (no smoothing)
```

When the dashboard sees `scOk:1, scLast:0`, the operator knows the latest poll just
failed even though the 5-poll consensus is still "up." Useful for catching emerging
problems before they cross the SMS threshold.

## What changed — Fix 14 (server)

### Server now parses and stores solar telemetry state

New transient (RAM-only, not persisted) fields on `ClientMetadata`:
- `solarTelemetrySeen` (bool — has the server ever observed solar telemetry for this client)
- `solarInitOk`, `solarCommOk`, `solarLastPollOk`, `solarSetpointsOk`, `solarEverOk`,
  `solarImplausible` (mirrors of `scInit` / `scOk` / `scLast` / `scSpv` / `scOkEver` / `scImpl`)
- `solarErrTag[8]`, `solarLastResMs`, `solarLastMaddr`, `solarLastOkAgoS` (per-poll details)
- `solarTelemetryEpoch` (when these fields were last updated)

`handleTelemetry` parses the new fields into these slots after the existing VIN block.

### Server `/api/clients?summary=1` emits a `sol` sub-object per client

```jsonc
"sol": {
  "init":  0 | 1,
  "ok":    0 | 1,
  "last":  0 | 1,
  "spv":   0 | 1,
  "ever":  0 | 1,
  "impl":  1,                // optional
  "err":   "to|crc|ida|ifu|?",  // optional
  "resMs": <uint16>,         // optional
  "maddr": <uint16>,         // optional
  "lastOkS": <uint32>,       // optional
  "epoch": <double>
}
```

### Dashboard dashboard readout

- **Main dashboard** (`/dashboard`): when a client status dot is expanded, the info line now shows `RS-485:` followed by a color-coded state (`OK` green, `last poll FAIL` amber, `DOWN` / `NEVER OK` / `init FAIL` red), plus the relevant context (`err=`, `Xms`, `reg=0x`, `[impl]`).
- **Site-config page** (`/site-config?site=X`): each client header now shows an `RS-485` pill badge alongside the existing OTA badge. Hovering the badge shows a tooltip with the same context. Pill is omitted entirely for clients that have not yet reported solar telemetry (e.g. older firmware or solar-disabled clients).

## Examples

A healthy SunSaver client now emits per telemetry:
```jsonc
{
  "fv": "2.0.53",
  "scInit": 1, "scOk": 1, "scLast": 1, "scSpv": 1, "scOkEver": 1
}
```
Dashboard renders: **RS-485: OK (setpoints verified)** in green.

A test-client SunSaver that booted with the bus disconnected:
```jsonc
{
  "fv": "2.0.53",
  "scInit": 1, "scOk": 0, "scLast": 0, "scSpv": 0, "scOkEver": 0,
  "scErr": "to", "scResMs": 1000, "scMaddr": 8
}
```
Dashboard renders: **RS-485: NEVER OK err=to 1000ms reg=0x8** in red.

A client with a transient SunSaver brownout (1 missed poll out of 5):
```jsonc
{
  "fv": "2.0.53",
  "scInit": 1, "scOk": 1, "scLast": 0, "scSpv": 1, "scOkEver": 1,
  "scErr": "to", "scResMs": 1000, "scMaddr": 9, "scLastOkS": 60
}
```
Dashboard renders: **RS-485: last poll FAIL err=to** in amber.

A client with a totally broken library init:
```jsonc
{
  "fv": "2.0.53",
  "scInit": 0
}
```
Dashboard renders: **RS-485: init FAIL** in red.

## Trade-offs

- **Telemetry size**: ~12 B happy-path, ~62 B failed-path. Within Notecard limits with room to spare.
- **Daily size**: ~50 B extra in the failure block; daily-only.
- **No persistence**: server's solar state is RAM-only and will reset on server reboot. Re-populated from the next client telemetry.
- **Backward compatibility**: server v2.0.53 with older clients omits `sol` from the API (graceful degradation, dashboard simply shows no RS-485 badge for those clients).
- **No new SMS triggers**: existing `alertOnCommFailure` pipeline unchanged.

## Verification

After flashing both client and server:
1. With SunSaver connected and healthy → dashboard shows **RS-485: OK** in green
2. Disconnect the SunSaver wires from the MRC-1 → dashboard transitions to **RS-485: last poll FAIL err=to** (amber) within one poll interval, then to **RS-485: DOWN err=to Xs** (red) once 5 consecutive failures accumulate
3. Reconnect → dashboard returns to **RS-485: OK** within one successful poll
4. Daily report after a day with mixed errors → `solar.merrTo`, `solar.merrCrc`, etc. fields visible in Notehub event payload
