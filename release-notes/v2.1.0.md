# v2.1.0 — Internally-Powered DAC Sensor Loop (PWM power gating retired)

**Client + Server release.** Common firmware bumped to 2.1.0 / build seq 280.
Flash both the client and the server. A v2.1.0 server can still manage older
clients (it keeps sending the fields they understand via legacy fallbacks), and
a v2.1.0 client still accepts configs saved by an older server — but the new
loop-power method only takes effect once the **client** is on 2.1.0.

This is a **minor-version** release because it changes how 4-20mA sensors are
physically powered and wired. See
[CODE REVIEW/CODE_REVIEW_07012026_MOSFET_GATING_FAILURE_AND_SOFT_RAMP_PROOF.md](../CODE%20REVIEW/CODE_REVIEW_07012026_MOSFET_GATING_FAILURE_AND_SOFT_RAMP_PROOF.md)
and
[CODE REVIEW/CODE_REVIEW_07012026_DAC_LOOP_POWER_INTEGRATION_PLAN.md](../CODE%20REVIEW/CODE_REVIEW_07012026_DAC_LOOP_POWER_INTEGRATION_PLAN.md)
for the full investigation that led here.

## Why: the P1-P4 transistor gating is retired

Since v1.9.22 the client powered 2-wire 4-20mA transmitters through a
high-side transistor output (P1-P4) on the Arduino Pro Opta Ext A0602, duty
cycling the loop supply between samples to save power. Bench forensics on the
test unit found:

- A **failed P1 gating MOSFET** (0V output under every drive method while
  P2-P4 delivered 12V), i.e. the gating output itself became the
  single point of failure for the whole measurement chain.
- The gating outputs are rated for only ~100mA and are vulnerable to inrush /
  inductive kick on long field cables; a hard 0→100% PWM step is exactly the
  stress profile most likely to kill them.
- A failed gate produces the confusing "valid-looking but stuck mA" family of
  symptoms that already cost several field-debugging cycles (see the
  v2.0.75-v2.0.77 framed-read hardening work).

## New method: the sensor channel powers its own loop

The AD74412R on the A0602 can drive a channel's DAC to ~11V **out the
channel's own +/- screw terminals** and simultaneously measure loop current on
the same pair (Blueprint `adding_adc` overlay). v2.1.0 uses this as the one
standard way to power a 4-20mA transmitter:

1. `SET CH_DAC` — channel becomes a voltage DAC, current limit disabled
   (allows up to 25mA loop current), no slew limiting.
2. `SET DAC` — output driven to 11.0V (raw 8191).
3. `SET CH_ADC` with `adding_adc=ENABLE` — a current ADC is overlaid on the
   same channel without disturbing the DAC drive.

All three frames are CRC-framed with full ACK validation, retried up to 3
times. After the configured warmup (default 3000ms) the existing framed +
CRC-validated read path (v2.0.75+) samples the loop 5 times, then the channel
is returned to high-impedance until the next duty cycle.

**Benefits**

- **Two wires, one terminal pair.** The transmitter is wired to the sensor
  channel's own I+/I- terminals — no separate power channel, no P1-P4 wiring,
  no power-channel dropdown in the config UI.
- **No gating MOSFETs in the path.** The failure-prone transistor outputs are
  bypassed entirely (the fried-P1 unit reads perfectly with the DAC method).
- **Duty-cycled power is preserved.** The loop is still only powered during
  sampling bursts, so the low-power profile is unchanged.
- **Pilot LED.** The channel's yellow face LED on the A0602 lights while the
  loop is powered and clears when it drops, so a technician can watch sensor
  power cycles at the cabinet.

**Bench verification (test Opta + A0602, Dwyer 626 0-5PSI transmitter on
channel 0 / terminals I1+ and I1-):** open loop reads 0.00mA (a true fault,
not a fabricated value); with the transmitter connected the loop read 3.69mA,
and after the transmitter's zero-point push-button calibration it sits at
4.004mA — a healthy live zero.

## Client changes

- `readCurrentLoopSensor()` now powers the loop via
  `tankalarm_configureDacLoopPowered()` (new in `TankAlarm_I2C.h`) instead of
  `tankalarm_setPwm()` on a gating channel, and powers down via
  `tankalarm_disableDacLoopPowered()` (channel → high-impedance).
- The channel LED is set/cleared with `tankalarm_setExpansionLeds()`
  (Blueprint `ARG_OA_SET_LED` 0x15) in lockstep with loop power.
- Config fields renamed: `pwmGatingEnabled/Warmup/SampleDelay` →
  `loopPowerEnabled/loopPowerWarmup/loopPowerSampleDelay`.
  **`pwmGatingChannel` is retired and ignored** — power comes from the
  sensor's own channel. Legacy `pwmGating*` keys are still parsed as
  fallbacks, so configs saved by an older server or an old on-flash
  config migrate automatically (enabled stays enabled).
- The configured warmup is now actually honored (previously hardcoded
  3000ms) and is chunked with watchdog kicks so a long warmup cannot
  starve the watchdog. The configured sample settling delay is honored too
  (previously hardcoded 300ms).
- The `CURRENT_INPUT_EXT_POWER` channel-function verification is removed from
  the loop-powered path (the channel legitimately reports a DAC function with
  an ADC overlay in this mode). Fault codes `pwm_nack` (1) and `func_wrong`
  (3) are retired but keep their enum slots; DAC power-up failures report
  `config_nack` (2).
- Loop Power **Disabled** (externally powered transmitter) now explicitly
  re-configures the channel as an externally-powered current ADC before every
  sampling burst — previously the non-gated path never configured the channel
  at all and could not produce a valid read after an expansion power cycle.
- The `pg` telemetry flag is kept and now reports the DAC loop-power
  configuration result.

## Server changes

- Client config editor: the **PWM Power Gating** enable and **Gating Output
  Pin (P1-P4)** dropdowns are removed. In their place a single **Sensor Loop
  Power** select (`Enabled (Powered from channel terminals)` /
  `Disabled (Externally powered)`) appears for 4-20mA sensors, defaulting to
  Enabled. The existing Pin/Channel selector is the only channel input —
  it now selects both the measurement and the power terminals.
- Warmup Delay and Sample Settling Delay fields are kept and shown while loop
  power is enabled.
- Saving emits `loopPowerEnabled/loopPowerWarmup/loopPowerSampleDelay`;
  loading accepts both the new keys and legacy `pwmGating*` configs.

## Wiring / migration for existing installations

1. Move the transmitter's two wires from the P-terminal supply arrangement to
   the sensor channel's own terminal pair (e.g. channel 0 = **I1+ / I1-**):
   transmitter **+** → I1+, transmitter **-** → I1-.
2. Remove any wiring to the P1-P4 gating terminals for that sensor.
3. Push the client's config from the v2.1.0 server config page (existing
   configs migrate: gating enabled → loop power enabled; the old gating
   channel is ignored).
4. Watch the channel's yellow LED blink during each sampling burst and
   confirm a plausible mA reading on the dashboard.

No Notehub / notefile schema change: telemetry keys are unchanged
(`ma`, `ma_raw`, `fault`, `pg`).

## Also in this release (common library)

- `tankalarm_configureDacLoopPowered()` / `tankalarm_disableDacLoopPowered()`
  — ACK-validated Blueprint framing for the internally-powered loop.
- `tankalarm_setExpansionLeds()` — A0602 front-cover LED bitmask control.
- `tankalarm_rampUpPwm()` / `tankalarm_rampDownPwm()` — soft-start/stop PWM
  helpers (10-step, ~1s ramps). Not used by the client anymore; retained for
  utilities and any future controlled use of the transistor outputs.
- New test utility `TankAlarm-112025-Sensor_Utility/A0602_DAC_Loop_Powered_Test`
  — standalone DAC loop bring-up / transmitter calibration sketch (live mA
  readout for the transmitter's zero-button procedure).
