# v2.1.1 — Hotfix: DAC loop-power apply-settle timing (v2.1.0 read_fail)

**Client + Common hotfix release.** v2.1.0's internally-powered DAC loop could
return `read_fail` / 0.000mA on every sample despite a healthy sensor. Flash
the client; the server build is unchanged apart from the version string.

## Root cause (bench-proven)

The A0602 expansion applies each Blueprint `SET` command in its own main
loop — the 4-byte ACK only means *"command queued"*, not *"applied"*.
`tankalarm_configureDacLoopPowered()` sent its three frames back-to-back
(~1ms apart):

1. `SET CH_DAC` (channel → voltage DAC)
2. `SET DAC` (11.0V value)
3. `SET CH_ADC` + `adding_adc` (current-ADC overlay)

At that spacing the 11V value in step 2 is processed **before the channel has
actually become a DAC**, so the value is dropped and the applied DAC drives
0V. Every subsequent read then returns a perfectly valid frame with `raw=0`
(0.000mA) — indistinguishable from an open loop, reported as `read_fail`
(all samples below live-zero).

An A/B falsification test on the bench unit
(`TankAlarm-112025-Sensor_Utility/A0602_DAC_RawFrame_Diag`, alternating
duty cycles on the same hardware) proved it:

| inter-frame settle | result |
|---|---|
| 0ms (v2.1.0 behavior) | valid frames, `raw=0`, 0.000mA forever |
| 300ms | live loop current immediately (10.2mA during transient, 4.0x mA settled) |

The managed-library utility never hit this because `OptaController.update()`
continuously re-pushes the DAC value, masking the lost write.

The bus-silence hypothesis (A0602 comms-timeout resetting the DAC during the
client's silent 3s warmup) was tested and **falsified**: with a settled
config, readings survive 3s and 8s of bus silence unchanged.

Also observed: the `GET_CHANNEL_FUNCTION` (0x40) answer buffer on current
A0602 firmware can return stale/garbage bytes. v2.1.0 already removed that
check from the loop-powered path; do not reintroduce it.

## Fix

`TankAlarm_I2C.h`: `tankalarm_configureDacLoopPowered()` now waits
`TANKALARM_DAC_APPLY_SETTLE_MS` (default 300ms, override-able) after the
`CH_DAC` ACK and after the `SET_DAC` ACK, so each stage is applied by the
expansion before the next is sent. Worst-case added latency per sampling
duty cycle: 600ms (before the existing warmup).

New diagnostic `TankAlarm-112025-Sensor_Utility/A0602_DAC_RawFrame_Diag`
(raw-frame hex forensics, alternating settle A/B, function readback) is
included for future A0602 protocol debugging.
