# v2.1.3 — PWM switched loop supply reinstated as a selectable mode (alongside DAC)

**Client + Server release. PWM path bench-validated 2026-07-02** on the test
Opta + A0602 (12V battery → Vpwm, sensor + → P2, sensor − → I1+): four+
duty cycles with soft ramp-up/down active, PWM ACK probe OK, ext-power ADC
config OK, 5/5 valid samples per burst averaging **4.525-4.528 mA** — matching
the DAC method's 4.516 mA on the same water column and the ~4.48 mA physics
prediction (three-way cross-validation of wiring, method, and scale). See
`TankAlarm-112025-Sensor_Utility/A0602_PWM_Loop_Test`.

## Why

The DAC internally-powered loop (v2.1.0) tops out at the AD74412R's 11V
output ceiling, and the loop drops ~2V across the module's internal 100Ω
sense resistor at 20mA — leaving ~9V at the transmitter at full scale, below
some transmitters' minimum (e.g. Dwyer 626: 10V). Installations on a 12-13V
battery-float (or 24V) rail need the higher external supply. v2.1.3 therefore
reinstates the P1-P4 transistor switched supply as a **selectable loop-power
mode**, with the DAC mode remaining the default.

## Loop power modes (server config → client behavior)

| UI selection | Wiring | Supply at transmitter (20mA worst case) | ADC scale |
|---|---|---|---|
| **Enabled — DAC 11V from channel terminals** (default) | 2 wires on the sensor channel's +/- | ~9.0V | bipolar ±25mA |
| **Enabled — PWM switched external supply (P1-P4)** | supply → P-terminal → transmitter + ; transmitter − → channel In+ | rail − ~2V (12.5V battery → ~10.5V; 24V → ~22V) | unipolar 0-25mA |
| **Disabled (Externally powered, always on)** | supply direct to transmitter, no switching | rail − ~2V | unipolar 0-25mA |

Both enabled modes duty-cycle the loop (powered only during sampling bursts)
and light the sensor channel's yellow LED while powered.

## Fixes applied to the formerly-retired PWM code (review findings)

The pre-2.1.0 implementation was **not** reinstated as-is. Four defects were
corrected:

1. **Hard 0→100% gate step** → replaced with the soft-start/soft-stop ramps
   (`tankalarm_rampUpPwm`/`tankalarm_rampDownPwm`, 10 steps ≈ 1s each way).
   The instantaneous step is the inrush/flyback stress profile implicated in
   the bench unit's P1 MOSFET failure; the ramps were built for this purpose
   but never wired into the client before the mode was retired. Failure
   paths also ramp down instead of snapping the gate off.
2. **`GET 0x40` channel-function verification removed** — v2.1.2 forensics
   proved the 0x40 answer buffer returns stale bytes on current A0602
   firmware; the old hard `func_wrong` fault could false-trigger. The framed
   read's CRC/channel-echo validation plus live-zero/over-range guards cover
   mis-configuration.
3. **Configured warmup and sample delay honored** (old code hardcoded
   3000ms/300ms), with the warmup chunked around watchdog kicks.
4. **Explicit PWM-accept probe**: the ramp steps are fire-and-forget, so one
   final ACK-checked full-on `SET_PWM` frame follows the ramp; a NACK raises
   the reinstated `pwm_nack` (1) fault and tears down cleanly.

The measurement scale is mode-aware (v2.1.2 lesson): PWM mode is an
externally-powered loop → unipolar reader; only the DAC mode uses the
bipolar loop-powered reader.

## Config schema

- New keys: `loopPowerMode` (`"dac"` | `"pwm"`), `loopPowerPwmChannel` (0-3
  = P1-P4), `sensorMinVoltage` (transmitter datasheet minimum, default 10.0V).
  Existing `loopPowerEnabled/Warmup/SampleDelay` unchanged.
- Migration: pre-2.1.0 configs carrying `pwmGatingEnabled`/`pwmGatingChannel`
  (and no `loopPower*` keys) map to **PWM mode with their original gating
  channel** — matching those installations' physical transistor wiring.
  v2.1.0-2.1.2 configs (no mode key) remain DAC. New sensors default to DAC.

## mA signal-clipping detection (the "silent failure" guard)

A supply-starved transmitter does not fail to 0mA — it **plateaus**. The loop
drops `I × 100Ω` across the module's internal sense resistor, so a sagging
battery caps the deliverable current at
`clipMa = (Vsupply − sensorMinVoltage) × 10 mA/V`. Example: an 11.5V battery
with a 10V-minimum transmitter clips at 15mA — a 90%-full tank that needs
18.4mA reads 15mA and silently charts as 68%.

Every successful sampling burst is now assessed against that ceiling:

- **Supply voltage source:** DAC mode uses the fixed 11.0V output; PWM mode
  uses `getEffectiveBatteryVoltage()` — **SunSaver MPPT via RS-485 Modbus
  preferred, analog Vin divider fallback** (strict priority, cache-only, no
  extra bus traffic). If neither source has data (e.g. RS-485 down and no
  divider fitted), the assessment is skipped and the operator relies on the
  existing stuck-reading detection.
- **Telemetry flags** (only on valid readings, never blocking them):
  - `vsup` — supply voltage used for the assessment;
  - `clip:1` — reading is at/above the computed ceiling (within 0.25mA):
    the level may actually be HIGHER than reported;
  - `lv:1` — supply below `sensorMinVoltage + 2.2V`, i.e. full-scale 20mA
    cannot be guaranteed right now (low readings are still trustworthy).
- **Serial diagnosis line** extended:
  `CL ch0: 15.02 mA (n=5/5, power=pwm, vsup=11.52, CLIP?)` plus an explicit
  WARNING line with the ceiling math when clipping is suspected.
- Honest corollary: in DAC mode (11V fixed) a strict-10V transmitter has a
  ~10mA ceiling — mid-scale — so such sensors report `lv:1` continuously on
  DAC power. That is the correct signal to move that installation to PWM
  (12-24V) or externally-powered mode.
- Server config page: new **Sensor Min Supply Voltage** field (per sensor,
  default 10V) feeds the ceiling computation.
- Server config page: live **"Usable range check" advisory** — as the
  installer selects the power mode and types the sensor's range and minimum
  voltage, the page computes the clip ceiling (DAC = 11V; PWM = a
  conservative 12.0V battery planning value) and translates it into
  engineering units: e.g. *"clips at ~15.0mA = 69% of span — readings above
  ~34.4 PSI (of 0-50 PSI) may be silently clipped. Tip: a 0-100 PSI sensor
  is trustworthy up to ~68.8 PSI."* Green when the full 4-20mA span is
  available; red when the supply cannot even sustain the 4mA live-zero.
  This makes the deliberate **over-ranged-sensor strategy** (installing a
  larger-range transmitter and using only its lower, un-clipped span) a
  first-class, guided option at configuration time.

## Server config page

- "Sensor Loop Power" select now offers the three modes above; the P1-P4
  "PWM Power Output" dropdown appears only in PWM mode.
- New always-visible (for 4-20mA sensors) **loop voltage check warning box**
  telling the installer about the ~2V full-scale drop across the internal
  100Ω sense resistor plus wiring, the per-mode supply arithmetic, and to
  verify the sensor's minimum supply voltage at its worst-case loop current.

## 12V vs 24V system classification (local-only) + actual voltage reporting

The Opta/A0602 supply spec is 12-24 VDC ±20% = **28.8V absolute maximum**. A
"24V" lead-acid bank exceeds that while charging (absorption 28.2-28.8V,
equalization 29.2-30.6V, plus RTS cold-weather temperature compensation) —
**24V installations must power the Opta through a DC-DC buck converter**,
now listed in the Bill of Materials.

The client classifies the bank from the live SunSaver battery voltage on
every successful Modbus poll (`> 17V` = 24V system) into
`SolarData.systemNominalV`, kept **local-only** for future protective logic
— deliberately not transmitted, to save bandwidth. The **actual battery
voltage** is already attached to reports whenever a source has data (and
skipped otherwise): telemetry `v`, daily report `v` + `vs` source tag, and
the daily solar block's `bv` — the server can derive the 12V/24V class from
those. No alert or SMS fires on this. Firmware does not write charger
setpoints over Modbus — the RAM setpoint registers are read-only copies,
the EEPROM custom block is unverified, writes wear EEPROM and require a
controller reset, and temperature compensation would defeat a voltage cap
anyway.
