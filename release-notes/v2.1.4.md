# TankAlarm v2.1.4 — Recovery-Heuristic Fix, Backoff Repair, Diagnostics Polish

**Release date:** 2026-07-02
**Build sequence:** 284
**Scope:** Client only (server unchanged, rebuilt for version consistency)

## Background

The v2.1.3 end-to-end bench verification (2026-07-02) surfaced a burst of seven
`diag.qo` `ev:"i2c-recovery"` events in a single Notehub session, all carrying the
signature `i2c_errs:0, cl_ok:0, cl_fault:6 (under_range), cl_dur_us:0`. Analysis
(see `CODE REVIEW/CODE_REVIEW_07022026_POST_CONFIG_DIAG_BURST_AND_CONFIG_LOSS.md`)
showed the I2C bus was perfectly healthy — the device was honestly reading 0.00 mA
because its stored config drove PWM loop power on a terminal with no sensor attached
(the new server config selecting the correct terminal had not yet arrived; the
client's inbound config poll runs on a 10-minute cadence on grid power).

The review also hypothesized the saved config had been "lost during OTA" because
telemetry carried factory-default names. That hypothesis was **falsified**: v2.1.1-era
telemetry carried the same default names, the boot log shows a successful config load
(no fallback messages), and the factory-default config (`sensorInterface=SENSOR_ANALOG`,
`currentLoopChannel=-1`) cannot produce current-loop sampling at all. The bench device
had simply never been given customer names before this test. No config was lost —
but the confusion showed the boot log needs to state what config is active.

## Fixes

### 1. Bus recovery no longer triggered by loop faults (the diag.qo flood)

`I2C_RECOVERY_SENSOR_ONLY` now fires only when the fault signature actually
implicates the bus/expansion (`pwm_nack`, `config_nack`, `read_fail`). When the
framed read passes CRC + channel-echo validation but reports `under_range` or
`over_range`, the loop itself is open/unpowered — SCL toggling cannot help, and
each attempt published a `diag.qo`. Operator visibility of an open loop is
unchanged: the sensor-fault `alarm.qo` and telemetry `fault:"under_range"` +
`ma_raw` still report it.

### 2. Recovery backoff and circuit breaker actually work now

Every recovery attempt reset the per-monitor `consecutiveFailures` counters "to
give the sensors a fresh chance" — which made `allCurrentLoopFailed` false on the
very next loop pass, landing in the branch that unconditionally reset the
exponential backoff to x1 and zeroed the attempt counter. Result: the backoff
could never grow and the `I2C_SENSOR_RECOVERY_MAX_ATTEMPTS` circuit breaker could
never trip (bench logs showed `backoff x1` on every attempt). The reset is now
gated on a `gCurrentLoopReadsOk` watermark: backoff/attempts only clear when a
framed read has actually succeeded since the last recovery.

### 3. `cl_dur_us` diag field wired up

`gLastClBurstMicros` was declared for the `diag.qo` `cl_dur_us` field but never
assigned — every diagnostic carried `cl_dur_us:0`. It now records the accumulated
on-the-wire framed-read time of the last sampling burst (settle delays excluded),
restoring the intended 400 kHz-window latency evidence.

### 4. Serial diagnostics: active config and PWM terminal now visible

- Boot and every server config apply print a `Config active:` summary — site name,
  monitor count, config epoch, and per-monitor sensor interface, current-loop
  channel, and loop-power mode (`dac` / `pwm@P<n>` / `ext`).
- The per-burst `CL ch<n>:` line now names the driven PWM terminal
  (`power=pwm@P2` instead of `power=pwm`), so a wrong-terminal config is visible
  at a glance.

## Field impact

- Devices with a genuinely open loop (sensor disconnected/unpowered) no longer
  burn cellular data on futile recovery cycles — one `alarm.qo` (rate-limited) and
  the regular telemetry fault fields carry the condition instead.
- Devices with a genuinely wedged bus recover exactly as before, but with working
  exponential backoff and a functional 10-attempt circuit breaker.
