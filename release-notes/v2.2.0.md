# TankAlarm v2.2.0

## Viewer: Server-Styled Dashboard + Contact Management

- **Dashboard redesign** — the viewer now uses the same visual language as the server dashboard: site-grouped sensor cards, a stats row (Sensors / Active Alarms / Summary Age / Next Fetch), and a firmware-update banner.
- **Active Alarms section** — when any sensor is in alarm, a copy of its card appears in a red-bordered section pinned to the top of the page; the section hides automatically when no alarms are active.
- **Request Update button** — asks the server for a fresh summary (`viewer_request.qo`); the server publishes immediately (rate-limited) and the viewer fast-polls its inbox so new data lands in seconds-to-minutes instead of the next 6-hour slot. The viewer Notecard now uses `hub.set sync:true` for prompt inbound delivery.
- **Viewer contact management** (`/contacts`) — viewer users can add/remove alarm notification contacts. Contacts sync to the main server (`viewer_contacts.qo`), are stored in the server contact directory with category `viewer`, and their phone numbers are auto-enrolled as SMS alert recipients. The server echoes the authoritative list back in every summary, so server-admin edits round-trip to the viewer. Viewer contacts show a **VIEWER** badge on the server's contacts page, where the admin retains full edit rights.

## SMS Pipeline Hardening (v2.1.6 changes, included in this release)

- **Offline alarm buffering** — client alarm/unload/solar/battery/power notes are no longer discarded when the Notecard is temporarily unavailable; they buffer to flash and flush on recovery.
- **Per-recipient SMS delivery** — the server now emits one `sms.qo` note per recipient with a scalar `to` field. A single Twilio route (`To Number = [body.to]`, `Message = [body.message]`) delivers to every configured recipient; the dashboard contact list is authoritative.
- **Per-contact alarm routing** — contacts with alarm associations receive only their selected alarms; contacts without associations receive everything.
- **Alarm re-notification** — while an alarm stays active, the server re-sends the alarm SMS every `smsReminderHours` (default 6 h, 0 disables) until it clears.
- **Test SMS** — new **Send Test SMS** button on the server settings page (`POST /api/sms/test`) verifies the Twilio route end-to-end.
- **System alarm fixes** — `battery_failure`, `solar_sunset`, and `i2c-error-rate` alarms are now classified correctly (no more phantom sensor-0 alarms; battery-failure SMS escalation honored).
- **Contacts data-loss fix** — saving from the `/contacts` manager page no longer wipes SMS recipients (merge-on-save + dangling-ID pruning).
- Clear/recovery SMS no longer consume the hourly alarm SMS budget; bounded retry on Notecard send failures; factory placeholder phone/email values no longer count as configured recipients.

## ⚠ Deployment Notes

1. **Notehub Route #4 (SMS/Twilio)** must be reconfigured when the server updates: set **To Number = `[body.to]`** and **Message = `[body.message]`**. The old `numbers[]` payload and the new route config are mutually incompatible — update both together.
2. **Notehub Route #1 (ClientToServerRelay)** needs two notefiles added: `viewer_contacts.qo` and `viewer_request.qo` (they originate from the viewer device — widen or duplicate the route if it is fleet-scoped).
3. See `Tutorials/Tutorials-112025/NOTEHUB_ROUTES_SETUP.md` for the updated route configuration.
