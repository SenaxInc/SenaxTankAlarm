# TankAlarm v2.2.12

Viewer networking overhaul: DHCP-first with a Starlink-safe static fallback, compile-time overrides for custom builds, and a server-side IP address selector delivered to the viewer over Notehub.

## Viewer

- **DHCP first, static fallback** — the viewer still requests a DHCP lease at boot (3 retries), but when no lease is granted it now falls back to a static profile instead of staying offline. The built-in fallback follows the Starlink router convention: **192.168.1.15**, gateway 192.168.1.1, subnet 255.255.255.0, DNS 1.1.1.1. Starlink assigns DHCP clients from 192.168.1.20–254, so addresses .2–.19 are never handed out automatically and cannot collide.
- **Custom-build overrides** — all network defaults can be pinned at compile time in `ViewerConfig.h` (`VIEWER_USE_STATIC_IP`, `VIEWER_STATIC_IP_1..4`, `VIEWER_GATEWAY_1..4`, `VIEWER_SUBNET_1..4`, `VIEWER_DNS_1..4`), matching the existing printer-config pattern.
- **Server-pushed network profile** — the viewer summary note now carries an optional `net` object. The viewer applies each profile revision exactly once, saves it to QSPI flash (`/vcfg/viewer_net.json`, partition 4 — same layout the client uses, other partitions untouched), restores it at boot before Ethernet bring-up, and re-initializes Ethernet immediately when a profile arrives mid-run. Because the summary arrives over **cellular**, a viewer with wrong LAN settings can always be re-addressed remotely; boards without provisioned storage self-heal from the next summary.

## Server

- **Viewer Network panel** (Server Settings → Viewer Device) — choose *DHCP (static fallback)* or *Static IP* and enter IP / gateway / subnet / DNS. Help text documents the Starlink-safe manual range. Saved values round-trip through `/api/server-settings` with full dotted-quad validation (malformed addresses are rejected with HTTP 400; static mode requires an IP).
- Profile changes stamp a new revision **only when something actually changed**, and the server publishes a fresh viewer summary immediately so the new addressing lands within the next Notehub sync instead of waiting for the 6-hour schedule.

## Upgrade Notes

- Flash both the **server** and the **viewer** binaries to use the push feature end-to-end. A v2.2.12 server with an older viewer is harmless (the viewer ignores the unknown `net` field); an updated viewer with an older server simply keeps its local defaults.
- The feature is dormant until you set a profile in Server Settings; leaving the fields blank keeps the viewer's own defaults.
- No client firmware changes (version bump only) and no Notehub route changes.
